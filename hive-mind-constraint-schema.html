§ 01
Real-World Constraint JSON Schema

The schema defines a RealWorldConstraint object. All fields marked required: true must be present for a constraint record to be routable. Fields marked required: false are enrichment fields — their absence triggers fallback logic defined in §04. The opacity_level field (1–5) scores how discoverable the constraint is to a standard analyst: 1 = publicly legislated and widely known, 5 = only discoverable through ground-level operational experience or primary source fieldwork.

§ 03
Routing Logic Matrix

Routing is determined by a two-axis decision: constraint_type × cost_floor_impact.severity. The matrix below defines which operator profiles receive alerts and at what cadence. A constraint must satisfy both axes to trigger routing — a low-severity physical_presence constraint routes differently than a critical one even though the type is identical.

Constraint Type Severity Inelastic? Captive Vendor? Target Profiles Alert Cadence Priority
proximity_distance critical Yes Yes real_estate_investor, hospitality_operator, reit_analyst, macro_analyst, labor_economist daily_digest P2
staffing_ratio critical Yes No labor_economist, policy_researcher, reit_analyst, venture_investor daily_digest P2
third_party_mandatory critical Yes Yes real_estate_investor, reit_analyst, macro_analyst, compliance_officer immediate P1
physical_presence high Yes No hospitality_operator, compliance_officer, labor_economist daily_digest P2
operating_days high Yes No real_estate_investor, hospitality_operator weekly_digest P3
licensing_person medium Partial No compliance_officer, hospitality_operator weekly_digest P3
operating_hours low Yes No hospitality_operator weekly_digest P4
reporting_interval low Yes No compliance_officer weekly_digest P4

Cross-sector suppression rule: When suppress_if_analog_exists: false (as in the Japan test vector), each analog instance routes independently. When set to true, the network suppresses duplicate alerts for the same constraint_type already delivered in the same jurisdiction within the prior 30 days, and instead appends to the existing constraint record's cross_sector_analogs array.

§ 04
Edge Cases and Fallback Extraction Logic
EC-01 · Data Conflict
Conflicting Jurisdiction Scope Claims
Scenario
A constraint is documented at national level (e.g., Private Lodging Business Act specifies "administrator required") but the 10-minute proximity standard only exists in municipal guidance — not in the national Act. The two sources appear to conflict: national law says "administrator," municipal guidance operationalises "within 10 minutes." Naive extraction treats these as separate or contradictory constraints.
Fallback Logic
Set scope: "municipal" (most restrictive level wins). Set confidence: "high" if both sources are primary. Populate legal_basis.secondary with the municipal document. Do not create two records — the municipal operationalisation is the binding one for investment analysis. Flag with opacity_level: 4 minimum when national and municipal specifications diverge.
Routing Impact
No routing change. Severity and type are determined by the binding (most restrictive) instrument.
EC-02 · Incomplete Data
margin_impact_pct Not Quantifiable
Scenario
A physical_presence constraint is identified and confirmed (e.g., a proximity requirement in a new jurisdiction) but no unit economics model has been run. The margin_impact_pct and cost_as_pct_gross fields cannot be populated with confidence. Omitting them leaves an incomplete record; guessing them degrades the schema's reliability.
Fallback Logic
Omit margin_impact_pct and cost_as_pct_gross (both are non-required enrichment fields). Set confidence: "medium". Set cost_floor_impact.severity: "high" as the conservative default when a captive vendor relationship is confirmed but cost is unquantified. Add to extraction_metadata a note: "unit_economics_model: pending". Route at P3 until model is complete, then re-route at confirmed priority.
Routing Impact
Downgrade to P3 / weekly_digest until margin_impact_pct is populated. Prevents noise alerts on unquantified constraints.
EC-03 · Stale Record
Regulatory Change After Extraction
Scenario
Japan's government signals it will accept robotic presence as partial substitution for the kaigo 1:3 staffing ratio — a precedent that, if formalised, would change technology_substitution.permitted from false to true and reduce cost_floor_impact.severity across the elder_care analog. The existing record RWC-JP-2018-001 does not reflect this change because review_due has not yet been reached.
Fallback Logic
The review_due field (set to six months post-extraction by default) acts as the scheduled stale-check trigger. For out-of-cycle changes: any network node may submit a constraint_amendment event referencing the original constraint_id. Amendment events do not overwrite the original record — they append a versioned amendments array. The routing engine re-evaluates priority on each amendment. Set confidence: "medium" on the amendment until the regulatory change is confirmed in primary legislation.
Routing Impact
Amendment triggers an immediate re-route to all original target_profiles regardless of alert_threshold cadence. A relaxation of a critical constraint is investment-relevant at P1 immediacy.
