§1Scope & Design Principles

This manual is a companion document to the Hive Mind Risk Routing Logic Specification v1.0. Where the spec defines expected-path behavior, this manual defines what the agent must do when inputs are contradictory, incomplete, or structurally outside the model's training envelope. Every edge case resolves to one of four deterministic outcomes — no ambiguous states are permitted.

Governing Principles

Three principles constrain all fallback logic in this document:

// PRINCIPLE 1: Conservative Dominance IF any_input_is_ambiguous_or_missing: ASSUME worst_case_value_for_that_input // Missing OI data = assume OI is elevated // Missing liq data = assume liq >= CASCADE_THRESHOLD // PRINCIPLE 2: Determinism Over Inference IF two_signals_conflict: DO NOT average_them() DO NOT weight_them() APPLY priority_ladder() // deterministic hierarchy, see §2 // PRINCIPLE 3: Human Override Points Are Sacred IF liq_4h_usd >= PAUSE_AGENT_THRESHOLD ($500M): PAUSE_AGENT() // requires explicit human RESUME signal NEVER auto_resume() // no timer, no condition-based resume
⚠ This manual does not override the base spec gate sequence (A1→A2→A3→B1→B2→C1→C2). Edge case logic fires before the gate sequence — it modifies input values or regime classification before gates are evaluated. Edge cases never bypass a gate; they set conditions that the gates then evaluate normally.

§2Conflict Resolution Priority Ladder

When signals conflict, this ladder defines which signal class wins. Higher priority always overrides lower priority. No exceptions.

  • P1Liquidation Volume (real-time) — Overrides all momentum signals. $150M+ halts regardless of EMA/RSI state.
  • P2Regime Gate (OI + Funding concurrent) — Overrides signal quality. R3 via Filter A halts even if B-gates would pass.
  • P3Data Integrity Checks — Stale or missing data defaults to conservative worst-case. Overrides any live signal derived from that source.
  • P4ATR / Volatility Regime — Overrides EMA-derived trending classification when ATR ≥ 3× avg.
  • P5EMA / RSI Momentum — Standard signal quality gates. Evaluated only after P1–P4 pass.

EC-01RSI / EMA Signal Divergence

EC-01
RSI / EMA Signal Divergence
EMA alignment suggests LONG while RSI is outside the valid long band, or vice versa. The momentum direction is internally inconsistent.
High
Trigger Conditions

EMA(21) > EMA(55) AND spread ≥ 1.2% (trending long signal) BUT RSI(14) is NOT in the 52–68 valid long band. OR: EMA bearish but RSI is 52–68 (momentum says long, trend says short).

Why It Happens

RSI mean-reverts faster than EMA crosses. During consolidation after a strong move, EMA spread remains wide while RSI compresses back to neutral. The two signals represent different time horizons of the same move.

Fallback Logic

// EC-01: RSI/EMA Divergence Resolution FUNCTION resolve_ema_rsi_divergence(ema_signal, rsi, direction): ema_aligned = check_ema_alignment(direction) // Gate B1 result rsi_in_band = check_rsi_band(rsi, direction) // Gate B2 result IF ema_aligned AND NOT rsi_in_band: // EMA says yes, RSI says no IF rsi > 75 OR rsi < 25: RETURN HALT // hard RSI block overrides bullish EMA (P5 < P5 band rule) ELSE: RETURN HALT // RSI out-of-band = no valid entry; wait for RSI to enter band IF NOT ema_aligned AND rsi_in_band: // RSI says yes, EMA says no RETURN HALT // EMA alignment required; RSI alone cannot open a position // Log: "EC-01 — EMA/RSI divergence, EMA not aligned" // Both agree → normal gate sequence applies, no edge case RETURN CONTINUE_TO_GATE_C
ℹ Developer note: This rule means B1 AND B2 must both pass independently. There is no weighted combination. The logic gates are sequential AND conditions, not a scoring system. A single gate failure terminates the sequence.

EC-02Missing or Stale Oracle Data

EC-02
Missing or Stale Oracle Data
CoinGlass OI, funding rate, or liquidation data is unavailable, delayed beyond the staleness threshold, or returns a null/error response.
Critical
Trigger Conditions

Any CoinGlass data source returns null, timeout, or HTTP error. OR: Last valid reading is older than 1 full 4H candle from the current evaluation timestamp.

Why It's Critical

During actual cascade events, CoinGlass and aggregator APIs frequently rate-limit or lag. The scenario most likely to produce missing data is the same scenario where the data matters most. Missing data is not a neutral signal.

Fallback Logic

// EC-02: Stale / Missing Data Resolution // Staleness threshold: last_reading_age > 4H (one full candle) FUNCTION check_data_freshness(source, last_ts, current_ts): age_hours = (current_ts - last_ts) / 3600 IF age_hours > 4.0: RETURN STALE IF source.response() IN [null, timeout, error]: RETURN MISSING RETURN FRESH FUNCTION apply_data_fallback(source_name, freshness): IF freshness IN [STALE, MISSING]: SWITCH source_name: CASE "OI": oi_vs_30d_ma_pct = +20 // assume elevated: triggers R2 minimum LOG_WARN("EC-02: OI data stale — defaulting to elevated (+20%)") CASE "FUNDING": funding_8h_pct = 0.06 // assume hot: triggers R2 minimum LOG_WARN("EC-02: Funding data stale — defaulting to hot (0.06%)") CASE "LIQUIDATIONS": liq_4h_usd = 200_000_000 // assume cascade: triggers R3 + Gate A1 FAIL LOG_WARN("EC-02: Liq data missing — defaulting to $200M (R3)") CASE "OHLCV": RETURN HALT_ALL // cannot compute EMA/RSI — full execution halt PAGE_HUMAN("EC-02: OHLCV feed unavailable — agent suspended") RETURN proceed_with_adjusted_inputs()

EC-03Flash Crash / Wick Spike

EC-03
Flash Crash / Wick Spike
A single 4H candle produces a price move exceeding ±8% from open to close, or a wick exceeding 12% of close price, that is not confirmed by the subsequent candle.
Critical
Trigger Conditions

abs(close - open) / open ≥ 0.08 in a single 4H bar. OR: (high - low) / close ≥ 0.12. These moves corrupt EMA and RSI for 1–3 subsequent candles even after price recovers.

Why It Corrupts Signals

EMA calculations incorporate the spike candle's close price, artificially inflating or deflating the spread for multiple bars forward. RSI overshoots into its hard block zones on the spike, then snaps back — creating a false signal inversion on recovery.

Fallback Logic

// EC-03: Flash Crash Detection + Signal Quarantine FUNCTION detect_flash_candle(candle): body_pct = abs(candle.close - candle.open) / candle.open wick_pct = (candle.high - candle.low) / candle.close IF body_pct >= 0.08 OR wick_pct >= 0.12: RETURN FLASH_DETECTED RETURN NORMAL FUNCTION apply_flash_quarantine(candle_idx): // Tag the spike candle and the 2 following candles as QUARANTINED FOR i IN [candle_idx, candle_idx+1, candle_idx+2]: candle[i].signal_quarantine = True // During quarantine: regime gates still evaluate (liq/OI/funding unaffected) // Only EMA/RSI-derived signals (Gates B1, B2) are suppressed SET_GATE_B1(FAIL, reason="EC-03 flash quarantine") SET_GATE_B2(FAIL, reason="EC-03 flash quarantine") // Existing positions: DO NOT close on flash alone // Maintain stops at pre-flash levels unless R3 independently triggered LOG_EVENT("EC-03: Flash candle quarantine active — B-gates suppressed for 3 candles") // On candle_idx+3: quarantine lifts automatically, resume normal evaluation
⚠ Quarantine does not cancel open positions. It only suppresses new entry signals. If a stop-loss level is hit organically during the quarantine window, normal exit logic applies. The quarantine is a new-entry gate, not a position management override.

EC-04Funding Rate Regime Inversion

EC-04
Funding Rate Regime Inversion
Funding rate crosses from strongly positive to strongly negative (or vice versa) within a single 8H window, while OI remains elevated. The prior regime classification is now directionally wrong.
High
Trigger Conditions

Funding rate change within one 8H interval exceeds 0.08% absolute (e.g., +0.05% → -0.03%). This is a delta of 0.08% in 8H, equivalent to a structural long-to-short regime flip in the perpetual market.

Why It's Dangerous

The Sep 2025 cascade (documented in the liquidation analysis) was triggered by exactly this pattern. Funding inverts because leveraged longs are forcibly exited, which then triggers the cascade that was hidden by the prior positive funding reading.

Fallback Logic

// EC-04: Funding Rate Inversion Detection FUNCTION check_funding_inversion(funding_current, funding_prev_8h): delta = abs(funding_current - funding_prev_8h) IF delta >= 0.0008: // 0.08% absolute change in 8H RETURN INVERSION_DETECTED RETURN STABLE FUNCTION resolve_funding_inversion(inversion, oi_elevated, direction): IF inversion == INVERSION_DETECTED: IF oi_elevated: // Elevated OI + rapid funding inversion = pre-cascade signature ESCALATE_TO_R3() // Force R3 regardless of current funding absolute value LOG_ALERT("EC-04: Funding inversion + elevated OI — R3 forced") RETURN HALT IF NOT oi_elevated: // Inversion without OI elevation = market repositioning, not cascade // Block new entries in prior regime direction for 2 candles SET_DIRECTIONAL_BLOCK(prior_regime_direction, candles=2) LOG_WARN("EC-04: Funding inversion detected — blocking prior direction for 2 candles") RETURN REDUCED_ENTRY // opposite direction still valid at 0.5× until regime restabilises
🔴 Elevated OI + funding inversion is the specific pre-cascade signature of the Sep 2025 event. This combination must force R3 regardless of what the absolute funding value is at the moment of evaluation. The inversion velocity is the signal, not the endpoint value.

EC-05Multi-Asset Cascade Correlation

EC-05
Multi-Asset Cascade Correlation
Asset B (SOL) independently shows clean entry conditions, but Asset A (BTC) is in R3 cascade. The correlated nature of crypto perpetual markets means SOL's signal is unreliable during BTC cascades.
High
Trigger Conditions

BTC-PERP is classified as R3 (via any filter) AND SOL-PERP evaluation is in progress. SOL's own OI/funding/liq data may appear normal — BTC cascade has not yet propagated.

Why It Fails Independently

BTC-SOL correlation on cascade days typically runs 0.85–0.95 within the same 4H candle. SOL liq data lags BTC liq data by 1–2 candles in most cascade events because the forced selling begins in BTC and rotates. Clean SOL data during a BTC cascade is an artifact of lag, not a genuine decorrelation.

Fallback Logic

// EC-05: Cross-Asset Cascade Propagation Guard GLOBAL ASSET_REGIMES = {"BTC": R0, "SOL": R0} FUNCTION evaluate_with_cross_asset_check(target_asset, signal): FOR peer_asset IN ["BTC", "ETH"]: // BTC checked first, highest market cap peer_regime = ASSET_REGIMES[peer_asset] IF peer_regime == R3_CASCADE_RISK AND peer_asset != target_asset: // Force target asset to R3 regardless of its own readings ESCALATE_TO_R3(target_asset) LOG_ALERT("EC-05: " + peer_asset + " in R3 — propagating cascade halt to " + target_asset) RETURN HALT IF peer_regime IN [R2_OVERHEATED, R4_VOLATILE] AND peer_asset != target_asset: // Downgrade target asset one regime step if own regime is R0 or R1 IF ASSET_REGIMES[target_asset] IN [R0_NEUTRAL, R1_TRENDING]: DOWNGRADE_REGIME(target_asset, R2_OVERHEATED) LOG_WARN("EC-05: Peer asset in R2/R4 — target downgraded to R2") RETURN proceed_with_adjusted_regime()

EC-06Post-R3 Clearance Race Condition

EC-06
Post-R3 Clearance Race Condition
R3 clearance conditions are met (liq below $50M for 2 candles, OI/funding normalized) but a new entry signal fires before the clearance counter has completed its second confirmation candle.
Medium
Trigger Conditions

R3 was active. Liq dropped below $50M on candle N. A valid LONG/SHORT signal fires on candle N+1 — before the second candle (N+2) has confirmed the clearance condition is sustained.

The Race

The spec requires 2 consecutive candles below threshold to clear R3. The race condition is when the signal arrives after candle 1 but before candle 2 has closed. If allowed, the agent would be entering on an unconfirmed clearance.

Fallback Logic

// EC-06: R3 Clearance Counter — Race Condition Prevention GLOBAL r3_clearance_counter = 0 // resets to 0 whenever any R3 condition re-triggers GLOBAL r3_active = False FUNCTION update_r3_clearance(liq_usd, oi_elevated, funding_hot): r3_conditions_met = (oi_elevated AND funding_hot) OR (liq_usd >= 150_000_000) IF r3_conditions_met: r3_active = True r3_clearance_counter = 0 // reset on any re-trigger RETURN IF r3_active AND NOT r3_conditions_met: IF liq_usd < 50_000_000: r3_clearance_counter += 1 IF r3_clearance_counter >= 2: r3_active = False r3_clearance_counter = 0 LOG_INFO("EC-06: R3 cleared after 2-candle confirmation") // Entry gate check: block if clearance counter is 1 (first clear candle only) FUNCTION check_r3_clearance_gate(): IF r3_active OR r3_clearance_counter == 1: RETURN HALT // still clearing — wait for second confirmation candle RETURN CLEARED

EC-07ATR Collapse / Dead Market

EC-07
ATR Collapse / Dead Market
ATR drops below 0.3× its 20-period average for 3+ consecutive candles. EMA spread may show a false trend because price is coiling without real momentum. RSI is stuck between 45–55.
Medium
Trigger Conditions

atr_vs_20avg_mult ≤ 0.3 for 3 or more consecutive 4H candles AND RSI is in the 45–55 range (neither long nor short band). The EMA spread may technically qualify for R1 but the price action is not directional.

Why It's an Edge Case

Low-ATR environments produce EMA spreads that look like trend but are noise. A position opened in this state will see its stop hit by normal bid/ask volatility before any directional move materialises. The spec's R1 classification depends on real trend energy, not stale EMA separation.

Fallback Logic

// EC-07: Dead Market / ATR Collapse FUNCTION check_atr_collapse(atr_mult_series): // atr_mult_series: list of last N candle ATR/20avg values collapse_candles = count([x for x in atr_mult_series[-3:] if x <= 0.3]) IF collapse_candles >= 3: RETURN COLLAPSE_DETECTED RETURN NORMAL FUNCTION apply_dead_market_rule(collapse, rsi, ema_spread_pct): IF collapse == COLLAPSE_DETECTED: IF 45 <= rsi <= 55: // RSI stuck in neutral + low ATR = no real trend // Downgrade R1 to R0 regardless of EMA spread OVERRIDE_REGIME(R0_NEUTRAL) SET_GATE_B1(FAIL, reason="EC-07: ATR collapse — EMA spread is noise, not trend") LOG_WARN("EC-07: Dead market detected — R1 downgraded to R0, no new entries") RETURN HALT IF rsi < 45 OR rsi > 55: // RSI has directionality even in low-ATR — allow but reduce size SET_CAPITAL_MULT(min(current_mult, 0.5)) LOG_WARN("EC-07: Low ATR, directional RSI — reducing to 0.5× pending breakout") RETURN REDUCED_ENTRY // Collapse clears when ATR returns above 0.5× 20-avg for 1 candle

§4Fail-Safe State Definitions

Every edge case resolves to one of these four named states. No other outcome codes are valid. Logging must record the state name, triggering EC code, and relevant input values.

StateEffectAuto-RecoveryHuman Required
HALT No new entries. Existing positions held with pre-event stops. Gate sequence exits immediately. Yes — on regime reclassification at next candle close (after clearance counter if R3) No (except PAUSE)
REDUCED_ENTRY Entry permitted at capital_mult capped at 0.5× or 0.25× as specified by the triggering EC rule. Gate sequence continues. Yes — on condition clearance No
FULL SUSPEND All new entries blocked. All automated monitoring paused. Human paged via alert channel. No agent actions until RESUME signal received. No Yes — explicit RESUME required
CONTINUE Edge case check passed. Proceed to next evaluation step. No modification to inputs or multiplier. N/A No
